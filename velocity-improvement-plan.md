## 핵심 진단: 현재 문서의 내러티브 문제

현재 문서는 **"개념 먼저 → 문제 나중"** 구조예요.  
독자가 왜 이 개념이 필요한지 느끼기 전에 정답(자동차 대시보드 3단 계층)을 먼저 보게 되어 흡수가 어려워요.

```
현재: [인트로] → [자동차 개요/3단계층] → [이동] → [속도] → [가속도] → [실습] → [요약(개요 반복)]
개선: [인트로+문제제기] → [이동] → [속도+왜필요한가] → [가속도+왜필요한가] → [실습] → [자동차개요(정리용)] → [요약]
```

### 🔵 섹션 0: 인트로 카드 — 재작성

**현재 문제:**
> "1편에서 화살표 키로 캐릭터를 움직였죠?"로 시작하는 건 좋은데, 바로 "속도와 가속도"라는 결론을 제목에서 먼저 노출해버려요.  
> 독자가 "왜 배워야 하는지" 느끼기 전에 배울 내용을 알아버려요.

**개선 방향:**
1편 코드를 직접 재현해서 "딱딱함"을 먼저 *체험*시킨 뒤, "이게 왜 이상하게 느껴지나요?"로 문제를 제기해요.
그리고 1편 vs 오늘의 차이를 **인트로에서 바로 보여줘요** (현재는 이 비교가 속도 섹션 중간에 묻혀 있음).

**추가할 내용 (Pinnguaq 교육 설계 참고, 원문 재작성):**
> 엔트리에서 유명한 게임들 — 마리오, 자동차 게임 — 을 떠올려봐요. 캐릭터가 키를 누르는 순간 최고속도로 달리다가 떼는 순간 바로 멈추나요? 아니에요.  
> *조금씩 빨라지고, 조금씩 느려져요.* 1편의 코드엔 그 '조금씩'이 없었어요.

**구체적 추가 요소:**
- 1편 코드 미니 재현 블록 + "오늘 코드" 미니 블록을 나란히 인트로에 배치
- "어떻게 다른지 느껴지나요? 그 비밀을 오늘 풀어봐요!" 로 마무리

### 🟡 섹션 1: 이동 — 소폭 개선

**현재 문제:**
> 내용 자체는 좋은데 마무리가 약해요. 섹션이 그냥 끝나버려서 다음 섹션과 자연스럽게 연결이 안 돼요.

**개선 방향:**
섹션 끝에 **브리지 질문** 하나 추가해요.

**추가할 내용 (원문 작성):**
> 💭 **생각해봐요:** 그런데 매번 딱 5칸씩 가는 게 왜 어색할까요?  
> 현실에서 자전거를 탈 때, 페달을 밟는 순간 바로 시속 20km가 되나요? 아니죠.  
> *처음엔 느리다가 점점 빨라지잖아요.* 그걸 가능하게 하려면 "이동하는 거리"가 변할 수 있어야 해요 →  
> 그게 바로 다음에 배울 **'속도'** 예요!

### 🟠 섹션 2: 속도 — 구조 개편 + Scratch Wiki 인용

**현재 문제:**
> - "속도" 정의가 나오기 전에 속도 비교 그림이 먼저 나와요 (설명 전에 그림이 오는 역순)
> - "1편 vs 2편" 비교 표가 여기 묻혀있는데, 이건 인트로로 올려야 해요
> - "방향도 들어있어요(+/-)" 설명이 갑자기 툭 튀어나와요 — 더 자연스러운 흐름이 필요해요

**개선 방향:**
1. 정의 먼저 → 그림 → 방향 개념 → 코드로 순서 재배치
2. Scratch Wiki 정의 인용 (CC BY-SA 4.0):

**Scratch Wiki에서 가져올 내용 (출처 표기):**
> *"Velocity is defined as speed in a given direction, and can, therefore, be negative, unlike speed alone (which is directionless and always positive)."*  
> — [Scratch Wiki, "Velocity"](https://en.scratch-wiki.info/wiki/Velocity), CC BY-SA 4.0

이것을 한국어로 변환해서 강조 박스로 추가:
```
📖 속도(Velocity)란?
"속도는 방향을 포함한 빠르기예요. 그래서 음수(-)가 될 수 있어요."
— Scratch Wiki (CC BY-SA 4.0) 참고
```

**Scratch Wiki 활용 예시 목록 (CC BY-SA 4.0, 한국어 번역 적용):**
속도 개념의 게임 적용 예시를 Scratch Wiki 기반으로 추가:
- 🏎️ 레이싱·드라이빙 게임 — 속도 변수로 훨씬 현실감 있는 조작감
- 🦘 플랫포머 게임 — 수직 이동(점프/낙하)에 특히 유용
- 🚀 우주 게임 — "아스테로이드" 같은 게임에서 관성 표현
- 🎱 물리 시뮬레이션 — 공이 튀기고 중력 받는 장면

### 🟢 섹션 3: 가속도 — 정의 보강 + Pinnguaq 어휘 참고

**현재 문제:**
> - "매 프레임마다 속도를 얼마나 바꿀까?"가 정의로 나오는데 조금 기계적이에요
> - 가속도가 *왜 필요한지* 설명하기 전에 숫자 표가 먼저 나와요

**개선 방향:**
Pinnguaq의 어휘 정의 방식을 참고해서 **실생활 연결 → 정의 → 숫자**  순으로 재배치해요.

**Pinnguaq 참고 어휘 정의 (자유 사용, 출처 표기, 원문 재작성):**
> - **Acceleration** — the rate of change of speed. Positive = speeding up, Negative = slowing down.  
> — [Pinnguaq, "Player control in Scratch"](https://pinnguaq.com/learn/player-control-in-scratch-2/) 참고

한국어 적용:
```
🦶 가속도(Acceleration)
매 프레임마다 속도가 얼마나 바뀌는지를 나타내요.
+ 값이면 점점 빨라지고, - 값이면 점점 느려져요.
```

**추가할 브리지 문장 (원문 작성):**
> 가속도를 0으로 만들면 속도가 그대로 유지돼요 — 우주에서 물체를 한 번 밀면 영원히 날아가는 것처럼요.  
> 그럼 게임에서 멈추려면 어떻게 해야 할까요? 👉 속도를 줄이는 힘이 필요해요.  
> 그게 바로 **마찰(Friction)** — 3편에서 다룰 내용이에요!

**Pinnguaq 참고 마찰 정의 (자유 사용, 출처 표기):**
> - **Friction** — resistance to movement that reduces speed.  
> — [Pinnguaq, "Player control in Scratch"](https://pinnguaq.com/learn/player-control-in-scratch-2/) 참고

미리보기 팁 박스로 추가:
```
🔮 3편 미리보기: 마찰(Friction)
속도에 0.9를 계속 곱하면 점점 줄어들어요.
속도 × 0.9 × 0.9 × 0.9 ... → 결국 0에 가까워져요!
```

### 🔴 섹션 4: 실습 — Scratch Wiki 코드 패턴 추가

**현재 문제:**
> 좋은 섹션이에요. 단 "실험해봐요" 이후에 더 발전된 코드 힌트가 없어요.

**추가할 내용 (Scratch Wiki CC BY-SA 4.0 기반):**
Scratch Wiki의 velocity 코드 패턴을 "보너스" 팁으로 소개:

```
⭐ 보너스: Scratch Wiki 방식 (CC BY-SA 4.0)
Scratch Wiki에서는 속도 감소를 이렇게 표현해요:
  '걷는속도'를 ('걷는속도' × 0.9) 로 정하기
이렇게 하면 키를 안 눌러도 서서히 느려져요 — 3편의 마찰 힌트!
```

출처: [Scratch Wiki, "Velocity"](https://en.scratch-wiki.info/wiki/Velocity) (CC BY-SA 4.0)
