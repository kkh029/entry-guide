디자인 가이드 (표현 — 색·아이콘·말투·전개)

작성일 2026-05-29 대상: docs/ 페이지의 표현(색·말투·전개) 결정·변경

페이지의 표현 층을 다룬다 — 같은 내용을 어떤 색·아이콘·말투·순서로 들려줄지. “무엇을 말하는가(내용)”는 authoring-editing-guide.html, “어디에 놓는가(구조)”는 new-page-guide.html가 담당한다.

TL;DR
  • 이 문서 = 표현(색·아이콘·말투·전개 순서). “맞는 내용인가(내용)” · “구조가 맞는가(레이아웃)”와는 다른 층이다.
  • 색 — 한 편 = 한 액센트 색으로 통일하고, 차분한 종이/잉크 팔레트 톤 안에서 인접 편과 구분되게 고른다. Entry 블록 색은 디자인 레버가 아니다(실제 에디터 색 고정).
  • 말투 — 해요체 중심, 격려, 이모지를 시각 앵커로. 명령조·딱딱한 설명체(-ㅂ니다)·어려운 한자어는 피한다.
  • 전개 순서 — 문제 먼저 → 개념 → 코드. 인트로 hook → 섹션 브릿지 질문 → 활동형 마무리.

이 문서의 경계 — 내용 · 레이아웃 · 디자인

세 가이드는 같은 페이지를 다른 층에서 다룬다. 헷갈리기 쉬우니 아래로 구분한다 — 한 항목이 어디에 속하는지 애매하면 “질문”을 기준으로 고른다.

층핵심 질문다루는 것문서
내용무엇을 말하는가 — 맞는가레퍼런스·출처/코드 주석·개념 정확성·대상 난이도·편집 절차authoring-editing-guide.html
레이아웃어디에 놓는가 — 구조가 맞는가HTML 뼈대·골격(header→stage→wrap→card)·네비·사이드바 TOC·색 토큰 인벤토리new-page-guide.html
디자인 (이 문서)어떤 느낌으로 — 표현이 맞는가색 선택·의미·아이콘/이모지·말투·콘텐츠 전개 순서·페이싱design-guide.html
겹치기 쉬운 세 지점 (확실히 구분)
  • 색 — “토큰이 무엇이고 어디에 선언/적용되나”는 레이아웃. “어떤 색을 골라 어떤 의미로 쓰나”는 디자인(이 문서).
  • 순서 — “섹션이 DOM에서 어떤 구조로 쌓이나”는 레이아웃. “문제→개념→코드처럼 어떤 리듬으로 전개하나”는 디자인.
  • 서술 — “사실이 정확한가·대상 수준에 맞나”는 내용. “말투가 친근한가”는 디자인.

D-1. 색과 아이콘

액센트 색 — 한 편 = 한 색

각 페이지는 고유 액센트 색 하나로 통일된다. 독자는 색만 보고도 “지금 같은 편을 보고 있다”를 느낀다. 헤더·.step-no 번호 원·.cta 버튼·section.card.c-{색}이 모두 그 한 색을 쓴다 (적용 메커니즘은 레이아웃 가이드, 여기서는 어떤 색을 왜).

레거시: 1~3편은 “레인보우”

1~3편은 섹션마다 색을 돌려쓰는(의미와 무관한 레인보우) 옛 방식이고, 4편은 혼합, 5편부터 단색으로 통일됐다. 신규 페이지는 5편식 단색을 따른다 — 헤더·CTA뿐 아니라 section.card도 에피소드 색 하나로. 기존 1~3편을 굳이 단색으로 되돌릴 필요는 없지만, 새로 만들 땐 색을 섞지 않는다.

페이지색토큰
1편 · 이동·충돌빨강--c1
2편 · 속도·가속도파랑--c2
3편 · 관성·마찰보라--c3
4편 · 포물선·발사주황--c4
5편 · 튕김·탄성청록(틸)--c5
엔트리란? (소개)초록--green
강조 한 점 (절제용)골드--accent

새 주제 색을 고를 때:

  • 앞뒤 편 색과 뚜렷이 구분되게 — 빨강·파랑·보라·주황·청록이 이미 쓰였으니 그 사이로 너무 비슷한 색은 피한다.
  • 전체 종이/잉크의 차분한 팔레트 톤에 맞춘다 — 형광·고채도 원색은 페이지 분위기를 깬다.
  • 고른 색은 assets/styles.css :root에 --c6 식으로 추가하고 헥스값을 등록한다(토큰 등록 절차는 new-page-guide.html).
Entry 블록 색은 디자인 레버가 아니다

블록 시각화의 색(--c-start 파랑, --c-flow 주황, --c-move, --c-judge, --c-data, --c-calc)은 실제 엔트리 에디터 색과 1:1로 고정이다. 페이지 액센트가 무엇이든 블록은 원색을 유지하고 탈색하지 않는다 — 학생이 에디터에서 같은 색을 보기 때문. 이건 표현 선택이 아니라 고정 규칙이라, 정식 위치는 레이아웃 가이드의 “공통 레이아웃 규칙” 표다.

아이콘 · 이모지

이 사이트의 “아이콘”은 대부분 이모지다. 색을 바꿀 수 없으니, 고르는 것이 곧 디자인이다.

  • 시각 앵커로 — 섹션 제목·팁 박스 앞에 주제를 대표하는 이모지 하나를 둬 시선을 잡는다.
  • 개념과 일관 — 같은 개념엔 시리즈 전체에서 같은 이모지를 쓴다(예: 🚗 자동차 비유, 🦘 점프, 🚀 우주, 🎱 물리, 🧭 방향).
  • 절제 — 모든 문장 끝에 붙이지 않는다. 한 단락 한 앵커 정도. 남발하면 앵커 기능을 잃는다.
  • 의미 있는 선택 — 🐢/🐰로 느림/빠름처럼, 이모지가 내용을 보조하게 한다(장식이 아니라).

대비·가독성

차분한 배경 위에서 액센트와 본문 대비를 충분히 둔다. 액센트 색 위 흰 글자(.cta 버튼)·본문의 진한 잉크색 등 기존 대비를 따른다. 색만으로 정보를 전달하지 말고(예: “빨간 게 정답”) 텍스트·이모지를 함께 쓴다.

D-2. 말투 (Voice & Tone)

대상은 초등·중학생이다. 같은 사실도 어떻게 들려주느냐로 흡수가 갈린다. (무엇을 말하느냐·정확한가는 내용 가이드 소관.)

  • 해요체 중심 — “~해요 / ~예요 / ~볼까요?”. 반말은 강조 한 줄에만 가끔.
  • 격려·친근 — 독자를 옆에서 거드는 말투. “함께 해봐요”, “어렵지 않아요”.
  • 질문형으로 끌어들이기 — “왜 그럴까요?”, “느껴지나요?”로 생각을 유도한다.
  • 이모지 앵커(D-1과 연동) — 감정·주제를 한 글자로 표시.

아쉬운 예 — 딱딱한 설명체

“속도는 방향을 포함하는 물리량입니다. 본 차시에서는 속도와 가속도의 개념을 학습합니다.”

→ 교과서 어투. 어리고 친근한 독자와 거리가 생긴다.

좋은 예 — 친근한 해요체

“속도엔 방향도 들어있어요! 🧭 숫자가 +면 오른쪽, -면 왼쪽으로 가요. 같이 확인해 볼까요?”

→ 말 걸듯 풀어 쓰고, 이모지로 시선을 잡는다.

피할 말투
  • 딱딱한 -ㅂ니다 설명체(“~입니다 / ~합니다”)로 일관하기.
  • 명령조 남발(“하시오 / 입력하라”) — 따라하기 단계는 “~해 봐요”로.
  • 학년 수준을 넘는 어려운 한자어·전문용어를 풀어 주지 않고 쓰기.

D-3. 콘텐츠 전개 순서

내용이 같아도 꺼내는 순서가 흡수를 가른다. 핵심 원칙은 문제 먼저, 정의 나중 — 왜 필요한지 느끼기 전에 정답부터 주면 딱딱해진다(velocity-improvement-plan.md의 핵심 진단).

  1. 인트로 hook — 앞 편 경험·익숙한 장면으로 “어? 왜 이렇지?”를 체험시킨다.
  2. 문제 제기 → 개념 — 그 어색함의 정체로 개념을 꺼낸다.
  3. 개념 → 코드 — 개념을 엔트리 블록·실습으로 옮긴다.
  4. 브릿지 질문 — 섹션/편 끝에 다음으로 넘어가는 질문 한 줄(“그럼 멈추려면?”)을 둔다.
  5. 활동형 마무리 — 같은 그림을 반복하는 요약 대신 “코드에서 찾아봐요” 같은 활동으로 닫는다.

아쉬운 흐름 — 결론부터

인트로(결론 노출) → 개요/3단 계층 → 이동 → 속도 → … → 요약(개요 반복)

→ 왜 배우는지 느끼기 전에 정답이 먼저. 요약이 개요와 중복.

좋은 흐름 — 문제 먼저

인트로+문제 제기 → 이동(브릿지) → 속도(왜 필요한가) → 가속도(왜) → 실습 → 정리 → 활동

→ 호기심으로 개념을 끌어내고, 마무리는 활동으로.

전개 순서 ↔ 내용의 연결

에피소드 순서는 load-bearing이다(2편이 1편의 “딱딱함”을 전제, 3편이 2편의 “안 멈춤”을 해결). 전개 리듬은 여기서, 그 브릿지가 사실로서 맞물리는지(내용 정합성)는 내용 가이드 PART B-3에서 점검한다.

디자인을 바꿀 때

표현 변경은 한 페이지에 갇히지 않는다 — 색·말투·전개는 시리즈 전체의 일관성과 직결된다.

  1. 목적 정의 — 무엇을 왜 바꾸나(가독성? 톤 통일? 색 충돌 해소?).
  2. 시리즈 일관성 점검 — 한 페이지만 색·말투가 튀면 시리즈가 어색해진다. 인접 편과 나란히 두고 본다.
  3. 색이면 — 인접 편 색·index.html의 .path strip 색과 충돌하지 않는지, 한 편 안에서 한 색으로 통일됐는지 확인.
  4. 전개·말투면 — 앞뒤 편 브릿지가 여전히 자연스러운지(리듬), 사실 정합성은 내용 가이드와 함께 확인.
  5. 미리보기 — python3 -m http.server 8000 --directory docs로 라이트/다크, 모바일 폭까지 본다.
