도트 캐릭터 스프라이트 도입

작성일 2026-05-30 대상: docs/ 전 페이지 헤더 + 1·2편 캔버스 데모

CSS 네모/canvas 도형으로만 그리던 캐릭터를 Zerie의 무료 픽셀아트 스프라이트(군인·오크)로 교체했다. 헤더는 편별 개념 모션을 유지한 채 비주얼만 스프라이트로, 1·2편 캔버스 주인공은 idle/walk 애니메이션 스프라이트로 바꿨다.

TL;DR
  • 에셋: Zerie Tiny RPG Character Asset Pack(무료, 사용·수정 OK·재배포 금지)에서 군인·오크 idle/walk 4시트만 가져와, 100×100 프레임을 캐릭터 크기로 타이트 크롭(군인 15×19·오크 22×16, 각 ~1KB) → docs/assets/characters/.
  • 헤더: 편별 개념 모션(점프·가속·미끄러짐·포물선·튕김)은 그대로 두고 비주얼만 스프라이트로 교체 + steps() 걷기 사이클 동시 적용. about-entry는 엔트리 로고라 제외.
  • 캔버스: 1·2편 drawHero()를 drawImage 스프라이트로(정지=idle·이동=walk·좌우 반전·로드 폴백). 물리 로직(heroW/heroH·이동·clamp·라벨)은 불변.
  • 라이선스: 필요 프레임만 포함 + 전 페이지 푸터 "도트 캐릭터 © Zerie" 링크 + CREDITS.txt.

개요

사이트는 빌드 단계 없는 정적 HTML이며, 그동안 모든 캐릭터(헤더 무대의 주인공, 인터랙티브 캔버스 데모의 주인공)를 CSS 박스와 canvas 프리미티브(사각형·원)로만 그려 왔다. 사용자가 받아 둔 도트 애니메이션 에셋을 활용해 (1) 헤더 캐릭터와 (2) 캔버스 데모 주인공을 실제 스프라이트로 교체하는 것이 목표였다. 엔트리 "모양 바꾸기" 교육 콘텐츠 추가는 이번 범위에서 제외했다.

에셋 준비 — 타이트 크롭

이슈

원본 시트는 프레임당 100×100이지만 실제 캐릭터는 그 안에서 매우 작고(군인 ~15×19, 오크 ~22×16) 투명 여백이 대부분이다. 이대로 쓰면 CSS/canvas에서 캐릭터를 적당한 크기로 키우려면 박스가 과도하게 커지고 배치(특히 발 위치)가 까다롭다.

해결

라이선스가 수정을 허용하므로, 모든 프레임에 걸친 alpha union bounding box를 구해 각 프레임을 캐릭터에 맞게 잘라 작은 시트로 재조립했다. 걷기 사이클 동안 캐릭터가 프레임 안에서 거의 제자리(발 baseline y=57 고정)라 잘림이 없다.

크롭 전 측정·재조립은 Pillow로 1회 처리했다(시트를 100px 슬라이스로 보고 union bbox 산출 → 각 프레임을 박스로 crop → 가로로 paste). 결과 파일은 모두 1KB 안팎으로 가벼워졌다.

헤더 적용 — 개념 모션 유지 + 스프라이트

발견

각 편 헤더 애니메이션은 단순 장식이 아니라 그 편의 물리 개념을 미리 보여주는 모션이었다(2편=가속, 3편=미끄러지다 멈춤+잔상, 4편=포물선, 5편=감쇠 튕김). 전부 "제자리 걷기"로 갈아엎으면 교육적 의미가 사라진다.

그래서 기존 위치 keyframe(개념 모션)은 그대로 두고, 비주얼만 스프라이트로 바꾸고, 걷기 프레임 사이클을 두 번째 애니메이션으로 동시 적용했다. 위치 keyframe은 left/transform만 건드리고, 걷기 사이클은 background-position-x만 건드리므로 충돌하지 않는다.

/* 예: 2편 — accel(가속) 위치 모션 유지 + 걷기 사이클 */
.hero{
  width:45px;height:57px;
  background:url(assets/characters/soldier-walk.png) 0 0/auto 57px no-repeat;
  image-rendering:pixelated;       /* 군인 원본이 오른쪽을 봄 → 반전 불필요 */
  animation:accel 3.2s cubic-bezier(.4,0,1,1) infinite,
            walkcycle .7s steps(8) infinite;
}
@keyframes walkcycle{to{background-position-x:-360px}}  /* 8프레임 × 45px */
크기·반전 규칙

헤더는 정수배 3× 스케일(군인 45×57, 오크 66×48)로 픽셀이 또렷하다. background:url() 0 0/auto 57px 로 높이를 고정하면 프레임 폭이 자동으로 45px이 되고, keyframe 종료값은 -(프레임수 × 폭).

방향 주의 — 군인·오크 모두 원본이 오른쪽을 본다(픽셀로 확인: 눈이 머리의 오른쪽. 오크는 메이스가 bbox를 오른쪽으로 늘려 헷갈리기 쉬우니 머리(녹색) 기준으로 판단). 따라서 오른쪽 연출이면 반전 없이 그대로 두고, 왼쪽이어야 할 때만 transform:scaleX(-1)로 뒤집는다 — 1·2·3·5편 군인·index 군인/오크는 모두 그대로(오른쪽), 4편 오크만 "왼쪽에서 날아오는 새"를 마주보도록 scaleX(-1)로 뒤집었다.

페이지유지한 개념 모션스프라이트 처리
index(신규) 좌→우 walk-across오크 + 군인 ×2 추격 퍼레이드(균등 간격)
1편 이동hop(점프, scaleX 반전 baked)군인 walk
2편 속도accel(가속)군인 walk + speedline 유지
3편 관성slide-stop(미끄러지다 멈춤)군인 walk + .trail 잔상을 군인 고스트로
4편 포물선fly(새 발사체, 유지)표적 돼지 → 오크 idle(왼쪽 보기로 날아오는 새를 향함)
5편 튕기기ball-bounce(공, 유지)공만 유지 — 추가 캐릭터 없음(공으로 충분)
about-entry—제외(엔트리 브랜드 로고, 캐릭터 아님)

캔버스 적용 — 1·2편 주인공

drawHero()의 그리기 부분만 스프라이트로 교체했다. 이동·충돌·라벨 등 데모 로직은 손대지 않아 학습 동작이 그대로다. (1편 소스 · 2편 소스)

Before — 도형

function drawHero(){
  const x=heroX, y=groundY-heroH;
  // 그림자 타원
  ctx.fillStyle='#ff6b5e';
  roundRect(x,y,heroW,heroH,10); ctx.fill();
  // 음영 패널 + 눈 2개(원)
  …
}

After — 스프라이트

function drawHero(){
  const moving = keys.left||keys.right;          // 2편은 Math.abs(vel)>0.3
  if(keys.right) facing=1; else if(keys.left) facing=-1;
  const set = moving ? SPR.walk : SPR.idle;
  // 그림자 타원(유지)
  const dw=FW*SCALE, dh=FH*SCALE, cx=heroX+heroW/2, dy=groundY-dh+3;
  if(!set.img.complete||set.img.naturalWidth===0){ /* 네모 폴백 */ return; }
  ctx.save();
  ctx.imageSmoothingEnabled=false;
  ctx.translate(cx,dy);
  ctx.scale(facing, 1);                          // 원본=오른쪽(facing 1=정, -1=반전)
  ctx.drawImage(set.img, (sprFrame%set.frames)*FW,0,FW,FH, -dw/2,0,dw,dh);
  ctx.restore();
}
영향 범위
  • 상태 추가: 이미지 프리로드(SPR.idle/walk), FW=15,FH=19,SCALE=3, sprFrame/sprTick/facing, img.onload=render.
  • loop()에 프레임 진행 한 줄: if(++sprTick%5===0) sprFrame++; (rAF ~60fps → ~12fps 걷기).
  • 판정: 1편은 키 입력, 2편은 vel 부호 → moving/facing. 정지 시 idle 시트.
  • 불변: heroW/heroH(물리 박스)·이동·clamp·xLabel/velLabel. 비주얼만 교체.
연결 — 4편/5편 캔버스는 왜 도형 유지?

4편 캔버스는 발사체 "새"가 도형으로 그려져 있어, 픽셀 오크만 넣으면 스타일이 섞인다. 오크는 4편 헤더에만 적용하고 인터랙티브 데모는 통일성을 위해 도형을 유지했다. 5편(튕기는 공)도 동일 취지로 캔버스는 그대로 두고 헤더에만 구경 군인을 추가했다. → 후속: next-steps "새 스프라이트화 검토".

유의

공개 GitHub Pages + 공개 소스 레포라 원본 PNG가 직접 다운로드 가능해지는 점은 "재배포" 회색지대다. 비영리 교육용 + 출처표기 + 최소 포함으로 통상 허용 수준이나, 추후 상업화·대량 배포 시엔 재검토 필요.

결과
  • 헤더: 로컬 서버 + 미리보기 스크린샷으로 7개 페이지 전부 확인 — 스프라이트가 지면 위, image-rendering:pixelated로 또렷, 의도한 방향. 2편에서 computed animationName="accel, walkcycle" 로 두 애니 동시 적용 확인.
  • 캔버스: 미리보기 탭이 백그라운드라 requestAnimationFrame이 throttle돼 라이브 캡처는 어려워, getImageData로 픽셀을 직접 분석 — 1편 504px·2편 486px의 캐릭터가 기대 좌표 중심(≈375/≈360)에 렌더링됨을 확인. 1편은 실행+우측 키로 x 증가(이동)도 확인.
  • 모든 assets/characters/*.png 참조가 실제 파일과 일치, 제거한 클래스(h-c1~3·헤더 .hero::before/after)의 잔존 참조 없음.
