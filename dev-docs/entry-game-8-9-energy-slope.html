8·9편 추가 — 에너지(롤러코스터) · 경사면과 마찰(눈썰매)

작성일 2026-05-31 대상: docs/entry-game-8-energy.html · docs/entry-game-9-slope.html 신규 + 7편 강등 + 공통 배선

외부 리뷰(review_20260529.md)의 후속 물리 주제 3·4순위 — 에너지(위치↔운동에너지 전환)와 경사면·마찰(비탈에서의 미끄러짐) — 을 8·9편으로 추가했다. 9편이 새 마지막 편이 되어 7편을 중간 편으로 강등했다. 1~7편과 동일한 「개념↔구현 분리」 표준 구조와 인터랙티브 캔버스 데모를 따른다.

TL;DR
  • 8편 에너지(인디고 --c8 #5a6fce·.c-indigo): 위치에너지→운동에너지 전환 + 마찰 손실. 데모 = 출발 높이 슬라이더로 굴러내려 목표 도달(높이 ↑ → 빠르고 멀리, 단 2배는 아님).
  • 9편 경사면·마찰(오키드 --c9 #b061c4·.c-orchid, 새 마지막 편): 경사가 미는 힘 vs 마찰이 버티는 힘 = 가속도. 데모 = 경사각 슬라이더 + 얼음/눈/모래 토글.
  • 내비 체인: 7편 「시리즈 완주」→「7편 완성+8편 브리지」로 강등, 8편→9편, 9편→목차. (6편 강등과 동일 패턴)
  • 정확성 가드: 8편은 v=√(2gh)의 비선형(높이 2배≠속도 2배)을 산문·.tip.warn으로 명시(4편 발사힘 보정과 동형), 9편은 진짜 공식 a=g·sinθ−μg·cosθ를 ⭐.tip.bonus로 분리.
  • 로컬 프리뷰 검증 완료 — 블록 SVG 8/8·콘솔 에러 0·데모 물리 시뮬·canvas 픽셀·네비 6→7→8→9→index.

개요

현재 시리즈는 이동·충돌(1) → 속도·가속도(2) → 관성·마찰(3) → 포물선·발사(4) → 튕김·탄성(5) → 힘·질량(6) → 운동량·충돌(7)로 이어진다. 리뷰가 제안한 추가 물리 주제 1·2순위(힘·질량=6편, 운동량·충돌=7편)는 PR #27로 완료됐고, 이번에 3순위 에너지와 4순위 경사면·마찰을 이어 8·9편으로 추가했다. 사용자가 "물리 주제 편 추가"를 명시해, 「점수·게임 규칙」(게임 완성도) 갈림길 대신 물리 흐름을 계속한다.

인터랙티브 데모 설계

두 데모 모두 7편 스캐폴드(.try-demo·.try-canvas-wrap.idle·HUD 2개·▶/↺·requestAnimationFrame·click-to-launch·drawGround)를 재사용하고 물리 코어와 컨트롤만 교체했다.

8편 — 출발 높이 → 굴러내려 → 어디까지?

언덕(직선 경사) 꼭대기 높이를 슬라이더로 정하고, 굴러내려 목표 🎯에 닿는지 본다. 언덕을 내려오는 동안 쌓인 낙하 속도가 그대로 수평 속도로 전환된다 — 이게 에너지 전환의 핵심.

// 언덕: 떨어지며 가속 → 바닥에서 수평 속도로 전환(에너지 전환)
if (phase === 'hill'){
  vy += GRAV; y += vy;
  x += vy * (160 / startH);          // 항상 바닥에서 x=rampBaseX
  if (y >= runwayY){
    y = runwayY; x = rampBaseX;
    vx = vy; vy = 0; phase = 'flat';  // 낙하 속도 = √(2·GRAV·startH) 가 수평 속도로
  }
} else if (phase === 'flat'){
  x += vx; vx *= FRICT;               // 마찰로 에너지 손실
  if (!reached && x + r >= goalX) reached = true;
  if (vx < 0.1 || x > W - r){ phase = 'done'; }
}
정확성 가드 — 높이 2배 ≠ 속도/거리 2배

바닥 도달 속도는 √(2gh)라 높이의 √에 비례한다. 시뮬레이션으로 확인: 높이 120→속도 10.39, 240→14.70 (1.41배, 2배 아님). 그래서 아동용 문장은 "더 높이 출발하면 더 빨라지고 더 멀리 간다"(단조 증가)만 단정하고, 비선형은 .tip.warn + 데모 HUD 숫자로 스스로 발견하게 한다. 4편 "발사힘 2배→훨씬 멀리" 보정과 동형(A-3).

9편 — 경사각 + 바닥 → 미끄럼 속도

경사각 슬라이더(10~45°)와 얼음/눈/모래 토글로 두 힘의 겨룸을 본다. 경사가 미는 힘은 진짜 sin으로 계산(4편 삼각함수 강화), 마찰은 시리즈 관례인 곱셈 감쇠로 흉내 낸다.

const slideGain = SLOPE_K * Math.sin(angRad);  // 경사가 미는 힘 ∝ sin(경사각)
vel += slideGain;
vel *= SURF[surface];                          // 바닥 마찰 (ice .99 / snow .96 / sand .92)
d += vel;                                       // 비탈을 따라 이동
정확성 가드 — 진짜 공식은 ⭐로 분리

진짜 비탈 가속도는 a = g·sinθ − μ·g·cosθ(미는 힘 − 버티는 힘). 데모는 경사항을 진짜 sin으로, 마찰항을 곱셈으로 흉내 내고, "곱셈은 진짜 −μg·cosθ가 아니라 흉내"임을 .tip.real이 명시. 공식 자체는 본문 산문이 아니라 ⭐.tip.bonus로 분리해 부담을 낮췄다. "2배 가파르면 2배 빠르다"는 금지(sin 비선형·도 단위≠기울기).

데모 물리 시뮬 결과 (의도한 순서 확인)

같은 각도에서 얼음≫모래(10°: maxVel 9.9 vs 2.0), 같은 바닥에서 가파를수록 빠름(모래 45°=8.0 vs 10°=2.0), 가장 빠른 조합 = 가파른 얼음(45°ice=16). 모래는 가파르게 해도 한계가 있음(45°sand 8.0 < 45°ice 16) — "마찰이 버틴다"는 본문과 일치.

렌더러 제약 대응

c-block 자식 안의 note는 누락된다

B-mini 렌더러는 c-block statements 내부 블록의 note를 그리지 않는다(렌더러 문서 참조). 그래서 반복문 안의 줄별 설명은 note 대신 코드 블록 바로 아래 .tip 산문(①②③)으로 옮겼다. 9편의 미끄럼속도 × 마찰(CALC reporter 안에 VARIABLE reporter 중첩)과 ⭐의 0.6 × (sin 경사각의 값)(이중 중첩)도 정상 렌더됨을 확인(블록 SVG 8/8).
