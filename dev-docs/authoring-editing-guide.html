페이지 작성·수정 가이드

작성일 2026-05-29 대상: docs/ 학습 페이지의 신규 추가·수정 작업

새 주제 페이지를 추가하거나 기존 페이지를 고칠 때 지켜야 할 콘텐츠 품질 원칙과 편집 절차를 정리한다. HTML 뼈대·레이아웃·복붙 체크리스트는 new-page-guide.html가 담당하고, 이 문서는 그 위에서 “무엇을, 어떻게 쓰고 고칠까”를 다룬다.

TL;DR
  • 새 페이지는 세 가지를 모두 갖춘다 — ① 신뢰도 있는 레퍼런스, ② 출처·코드 주석, ③ 대상 독자(초등·중학생) 수준에 맞는 정확한 내용. 말투·색·전개 같은 표현은 design-guide.html.
  • 수정은 ① 목적을 먼저 정의하고 → ② 현재 페이지의 레이아웃·중복을 점검하고 → ③ 다른 페이지와의 연관성(에피소드 순서·브릿지·footer/CTA)을 함께 검토한다.
  • 공통 레이아웃 규칙(골격·액센트 색·콜아웃·Entry 블록 색·네비·footer·폰트)은 추가·수정 모두 동일하게 따른다 — 정본은 CLAUDE.md “Page architecture”와 new-page-guide.html.
  • 외부 출처의 라이선스 제약은 velocity-improvement-plan.md가 SSOT다 (Scratch Wiki = CC BY-SA 4.0, Pinnguaq = 재작성+출처, Breakout Mentors = 아이디어만).
  • 마무리는 항상 CLAUDE.md의 PR 절차 — session-doc → fix-history → next-steps → origin/main 리베이스 → PR.

개요

이 사이트는 빌드 툴이 없는 정적 HTML 묶음이고, 각 페이지는 자기완결형이다. 그래서 작업은 항상 둘 중 하나다 — 새 주제 페이지를 추가하거나, 기존 페이지를 수정하거나. 두 작업은 손대는 곳이 다르지만, 마지막에 같은 PR 절차로 수렴한다.

PART A · 새 주제 페이지 추가하기

새 페이지의 목표는 “정확하고(레퍼런스) · 추적 가능하며(주석) · 대상 수준에 맞는(적정 난이도)” 세 가지다. 아래 세 절은 각각 그 하나씩을 다룬다. (이 세 가지를 어떤 말투·색·순서로 표현할지는 design-guide.html.)

A-1. 신뢰도 있는 레퍼런스 찾기

학습 콘텐츠는 “느낌”이 아니라 근거 위에 써야 한다. 개념 정의·공식·용어는 아래 신뢰 우선순위에 따라 출처를 고르고, 가능하면 둘 이상에서 교차 확인한다.

우선순위출처 유형예시활용
1순위엔트리 공식playentry.org, docs.playentry.org블록 이름·동작·UI는 반드시 공식 기준으로 확인
2순위공신력 있는 위키·교육기관Scratch Wiki, Pinnguaq, 네이버 커넥트재단개념 정의·교육 설계·활용 예시 (라이선스 확인 필수)
3순위개인 블로그·영상교사·개발자 블로그 등아이디어 참고만. 사실 주장은 1·2순위로 교차검증
피함출처 불명·AI 생성 떠도는 글—인용 금지. 사실 확인이 안 되면 쓰지 않는다

라이선스 먼저 확인

출처를 찾았으면 인용하기 전에 라이선스를 본다. 이 프로젝트에서 이미 쓰는 출처들의 제약은 velocity-improvement-plan.md에 정리돼 있고, 그게 SSOT다.

출처라이선스허용 범위
Scratch Wiki — VelocityCC BY-SA 4.0정의·코드 패턴·예시 직접 인용 가능 (출처 표기 필수)
Pinnguaq — Player control in Scratch자유 사용 (출처 표기, 표절 금지)설계 방식·어휘 정의·서술 구조 참고. 원문 재작성 후 출처 표기
Breakout MentorsAll Rights Reserved아이디어만 참고. 직접 인용·번역 인용 금지
하지 말 것

All Rights Reserved 문서를 그대로 번역해 옮기는 것은 저작권 침해다. 또한 라이선스가 자유롭더라도 출처 없이 가져오면 안 된다. CC BY-SA 자료를 인용할 땐 출처 표기는 물론, 우리 페이지의 라이선스 (CC BY-NC-SA 4.0)와의 양립성도 신경 쓴다 — 직접 인용 문장은 “인용”임이 드러나게 박스로 감싸고 출처를 붙인다.

A-2. 주석 달기

여기서 “주석”은 세 가지를 포함한다 — 독자에게 보이는 출처 주석, 정리용 참고 자료 박스, 그리고 다음 사람이 보는 코드(HTML) 주석. 셋 다 “이 내용이 어디서 왔고, 이 구조가 무슨 뜻인지”를 추적 가능하게 만든다.

① 출처 주석 (인라인)

외부 정의·문장을 가져온 자리에는 바로 옆에 출처를 단다. 기존 페이지는 인용 박스 (.tip.bonus) 안에 번역·재작성한 문장을 넣고, <small>로 출처 링크를 붙이는 패턴을 쓴다. 

<div class="tip bonus">
  <b>📖 속도(Velocity)란?</b><br>
  "속도는 <b>방향을 포함한 빠르기</b>예요. 그래서 <b>음수(-)가 될 수 있어요.</b>"<br>
  <small style="opacity:.7;">— <a href="https://en.scratch-wiki.info/wiki/Velocity"
     target="_blank">Scratch Wiki, "Velocity"</a></small>
</div>

② 참고 자료 박스 (페이지 하단)

footer 바로 위에 .meta-box로 그 페이지가 참고한 출처를 모아둔다. 인라인 주석이 “어느 문장”인지를, 이 박스가 “페이지 전체”의 출처를 보여준다. 

<div class="meta-box">
  <b>📚 참고 자료</b><br>
  • <a href="https://en.scratch-wiki.info/wiki/Velocity" target="_blank">Scratch Wiki, "Velocity"</a><br>
  • <a href="https://pinnguaq.com/learn/player-control-in-scratch-2/" target="_blank">Pinnguaq, "Player control in Scratch"</a>
</div>

③ 코드(HTML) 주석 — 다음 사람을 위해

빌드 툴이 없으니 다음 작업자는 원본 HTML을 직접 읽는다. 긴 페이지는 주요 섹션 앞에 <!-- 섹션명 --> 라벨을 달아 흐름을 한눈에 잡게 한다 (about-entry.html이 이 방식). 

<!-- 3) 주요 특징 -->
<section class="card c-blue">
  <div class="step-no">3</div>
  <h2>...</h2>
</section>

학습자에게 보여주는 블록 설명용 주석이 필요하면 Entry 블록 시각화의 .note 프리미티브를 쓴다 (코드 주석과 혼동하지 말 것 — 이건 화면에 보이는 학습 보조 텍스트다).

주석 작성 시 주의
  • 인용문은 반드시 한국어로 번역·재작성한다. 원문 영어를 그대로 붙이지 않는다.
  • 모든 외부 링크에는 target="_blank"를 붙인다(가능하면 rel="noopener"도).
  • 인라인 출처와 하단 .meta-box의 출처 목록이 서로 어긋나지 않게 한다.

A-3. 대상 독자에게 맞는 내용 고르기

독자는 초등·중학생이다. “이해하기 쉽다”를 내용 차원에서 이렇게 만든다 (같은 내용을 어떤 말투·순서·색으로 표현할지는 design-guide.html 소관).

  • 한 번에 한 개념 — 한 섹션에서 새 개념은 하나만. 곁가지는 다음 편으로 미룬다.
  • 정확하되 쉬운 용어 — 어려운 말은 풀어 쓰되, 사실은 A-1 레퍼런스로 확인한다. 쉽게 쓰려다 틀리지 않는다.
  • 난이도 적합성 — 학년 수준을 넘는 수식·전문용어는 비유로 대체하거나 “보너스”로 분리한다.
  • 선행 지식만 전제 — 앞 편에서 배운 것까지만 가정한다(에피소드 순서는 load-bearing). 연결의 내용 정합성은 PART B-3에서 점검.
표현은 디자인 가이드로

말투(해요체·이모지), 전개 순서(문제 먼저 → 개념 → 코드), 비유·브릿지 질문, 색·아이콘 선택은 design-guide.html가 정본이다 — 이 문서는 무엇을, 그 문서는 어떻게 들려줄지를 다룬다.

A-4. 새 페이지 작업 순서

  1. 레퍼런스 수집·라이선스 확인 — A-1 기준으로 출처를 모으고, 직접 인용 가능 여부를 먼저 판단한다.
  2. 뼈대 복사 — new-page-guide.html의 템플릿으로 파일을 만들고, 공통 레이아웃 규칙(골격·색·네비·footer·폰트)과 파일명 규칙을 지킨다.
  3. 콘텐츠 작성 — A-3(대상 수준에 맞는 정확한 내용)으로 무엇을 쓸지 정하고, 말투·전개 순서·색은 design-guide.html를 따른다.
  4. 주석·출처 — A-2의 인라인 출처 + .meta-box + 섹션 HTML 주석을 단다.
  5. 연결 — docs/index.html의 .toc-grid에 카드를 추가하고, 학습 흐름이 바뀌면 .path strip도, 이전 편의 .cta 버튼도 새 페이지를 가리키게 한다.
  6. 미리보기 — python3 -m http.server 8000 --directory docs로 열어 목차·CTA·사이드바 TOC가 동작하는지 확인한다.

PART B · 기존 페이지 수정하기

수정에서 어려운 건 “고치는 것”이 아니라 주변을 깨뜨리지 않는 것이다. 페이지 하나는 다른 페이지·공유 자산과 얽혀 있으므로, 손대기 전후로 아래 세 가지를 점검한다.

B-1. 목적을 먼저 정의

무엇을, 왜 바꾸는지 한 문장으로 적고 시작한다 — 오타 수정인지, 설명 보강인지, 구조 개편인지에 따라 점검 범위가 달라진다. (예: 2편 구조 개편은 velocity-improvement-plan.md를 SSOT로 삼아 그 계획을 따른다.)

B-2. 현재 페이지 점검 — 레이아웃 · 중복

레이아웃 점검
  • 위 공통 레이아웃 규칙(골격·콜아웃·네비·footer·폰트)을 그대로 따르는가? 기존 구조(section.card + .step-no, .tip, .cta, 사이드바 .toc-sidebar)를 임의로 바꾸지 않았는가?
  • 새 스타일을 인라인 style=로 남발하지 않았는가? 페이지 전용 규칙은 <head>의 <style>에 둔다.
  • 모든 section.card에 <h2>가 있는가? (사이드바 TOC가 이를 자동 수집한다.)
중복 점검
  • 같은 설명·그림이 페이지 안에서 반복되지 않는가? (2편은 ‘개요’와 ‘요약’에 같은 계층 그림이 중복됐던 사례가 있다 — 한쪽은 활동·요약으로 바꾼다.)
  • 여러 페이지에 공통인 스타일을 그 페이지에만 또 정의하지 않았는가? 공통은 assets/styles.css에 있어야 한다.
  • 추가하려는 설명이 다른 편에 이미 있는 내용은 아닌가? 그렇다면 다시 쓰지 말고 그 편을 가리킨다.

B-3. 다른 페이지와의 연관성 검토

에피소드 순서는 load-bearing이다 — 2편은 1편의 “딱딱함”을 전제로 시작하고, 3편은 2편의 “안 멈추는” 문제를 해결한다. 한 편을 고치면 앞뒤 편으로 가는 다리(브릿지)가 여전히 말이 되는지 확인한다.

점검 대상무엇을 확인하나

공유 자산 변경은 전 페이지에 영향

assets/styles.css의 토큰·공통 규칙을 한 페이지가 필요해서 바꾸면 모든 페이지가 영향을 받는다. 한 페이지만의 변경이면 그 페이지 <style> 안에 두고, 공통 변경일 때만 styles.css를 건드리되 다른 페이지를 모두 다시 확인한다. Entry 블록 색(--c-start 등)은 어떤 경우에도 탈색하지 않는다.

B-4. 수정 작업 순서

  1. 목적 한 줄 정의 — 무엇을 왜 바꾸는지.
  2. 대상 페이지 정독 — 손대기 전 현재 구조·중복 여부를 먼저 읽는다.
  3. 영향 페이지 목록화 — 앞뒤 편, index.html, 공유 자산 중 무엇이 얽히는지 적는다.
  4. 수정 — 기존 레이아웃·톤을 따르며 변경한다.
  5. self-review — B-2(레이아웃·중복) · B-3(연관성) 표를 다시 훑는다.
  6. 연결 동기화 + 미리보기 — 얽힌 페이지를 함께 고치고 로컬 서버로 흐름을 확인한다.

마무리 — PR 전 공통 절차

새 페이지든 수정이든, PR 올리기 전에 CLAUDE.md의 절차를 그대로 따른다.

  1. 세션 문서 — /session-doc으로 이번 변경을 dev-docs/에 기록(또는 기존 문서 갱신).
  2. 변경 로그 — fix-history.md 최상단에 한 줄 항목 추가(오타·주석·포매터만 고친 경우는 면제).
  3. 다음 단계 — next-steps.md에서 완료 항목을 정리하고 새 follow-up을 적는다.
  4. main 반영 — git fetch origin && git rebase origin/main 후 git log HEAD..origin/main이 비어 있는지(behind 0) 확인.
  5. PR 생성 — 위 문서들을 같은 PR에 포함한다. main은 절대 force-push 하지 않는다.
커밋·푸시 전

git 작업 전에는 git-security-check 스킬로 민감 정보·불필요한 생성 파일이 섞이지 않았는지 점검한다.
