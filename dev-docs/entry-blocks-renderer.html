엔트리 블록 SVG 미니 렌더러 (B-mini)

작성일 2026-05-28 관련 PR #16 대상 컴포넌트 docs/assets/entry-blocks/

교육 문서의 엔트리 블록 시각화를 div+CSS 흉내내기에서 실제 엔트리 에디터와 거의 동일한 SVG로 교체. renderer.js 단일 파일(~600줄)로 7개 모양 · 5개 필드 · 재귀 nesting 지원.

TL;DR
  • EntryJS 전체(17MB raw, 3.3MB gzip)는 정적 문서에 과해 도입 불가. 대신 데이터(색·아이콘)만 vendoring + 모양 SVG path는 자체 작성하는 B-mini 방식 채택
  • 지원 모양 7종(hat/stack/cap/reporter/boolean/cblock/cblock_double) + 필드 5종(text/input/dropdown/icon/block) + block 필드로 자식 블록 재귀 nesting
  • HTML <div class="entry-code" data-blocks='[...]'>로 선언, 페이지 로드 시 자동 SVG 변환
  • 1편/2편/3편의 코드 블록 12 묶음 + 색상 가이드 전체 교체. 옛 CSS 약 45줄 제거

배경

기존 학습 문서는 엔트리 블록을 <div class="eb hat"> / <div class="eb cblk flow"> 같은 자체 CSS 클래스로 흉내냈다. 컬러와 모서리 둥글기만으로 만든 사각형이라 실제 엔트리 에디터의 모양과 거리가 있었고, 텍스트·입력칸·c-block 들여쓰기 처리도 매번 수작업이라 코드 수정 부담이 컸다.

이슈
  • 옛 디자인: 스크래치풍 색(#5cb1ef 등) — 실제 엔트리(#00b400 등)와 시각적 불일치
  • 외곽 모양: 둥근 사각형뿐이라 hat·c-block의 ㄷ자·boolean의 육각형 등 형태 다양성 부족
  • 블록 작성 시 <div class="eb"><span class="bool">...</span></div> 식의 verbose HTML — 학습 작성자 부담
  • 학생이 엔트리 에디터를 직접 보면 우리 문서의 색·모양과 매칭이 안 됨

대안 검토

EntryJS(entrylabs/entryjs, Apache 2.0)의 실제 블록 렌더링 코드를 활용하는 방향을 우선 검토했다.

대안접근크기노력판단
A1. entry.min.js 통째 vendoring EntryJS 빌드 산출물을 그대로 로드, Entry.RenderView API 호출 17 MB raw · 3.3 MB gzip 적음 ❌ 교육 문서 한 페이지 위해 3 MB 다운로드는 과함
A2. webpack tree-shaking RenderView 진입점만 import하는 mini-entry 빌드 예상 수백 KB 중간 ❌ Entry가 글로벌 Entry.X 패턴이라 tree-shaking 효과 제한적, 위험 큼
B-mini. 데이터 vendoring + 자체 SVG 색·아이콘·라벨만 EntryJS에서 가져오고 모양 SVG path는 자체 작성 ~50 KB 큼 ✅ 채택
채택 — B-mini
  • 데이터 부분은 entrylabs/entryjs (Apache 2.0)에서 vendoring: 카테고리 색상(EntryStatic.colorSet) · 카테고리 마커 SVG 16종
  • 모양 부분은 production 엔트리 에디터(playentry.org/ws)의 시각적 디자인을 기준으로 SVG path 자체 작성. master의 skeleton/basic/basic_event.ts(옛 hat 안테나)와 다른 production 디자인(둥근 사각형 + 아이콘) 그대로 재현
  • 출처 고지: docs/assets/entry-blocks/NOTICE.md

모양 카탈로그

지원 7종. 모두 왼쪽 평평 + 오른쪽 둥근 캡슐(rx=18) 기본형에 위/아래 connector 변형을 더한 형태. SHAPE_DEFS 객체에 등록된다(renderer.js:L237-L244).

shape용도위 connector아래 connector본체
hat시작 이벤트 (when_run_button_click 등)평평▽ 볼록높이 36 · 왼쪽 평평 + 오른쪽 캡슐
stack일반 명령 (move·set_var 등)∇ 오목▽ 볼록높이 36 · 동일
cap마무리 (코드 멈추기 등)∇ 오목평평 (다음 블록 불가)높이 36 · 동일
reporter값 (변수·계산 결과)——높이 28 · 양옆 캡슐 (rx=14)
boolean판단 (참/거짓)——높이 28 · 양옆 뾰족한 육각형
cblock제어 단일 슬롯 (반복하기·만일~라면)∇ 오목▽ 볼록top(36) + 슬롯(들여쓰기 16) + bottom(12)
cblock_double제어 이중 슬롯 (만일~아니면)∇ 오목▽ 볼록top + 슬롯1 + middle(36) + 슬롯2 + bottom

예: hat 모양 path 생성 함수 (renderer.js:L248-L266):

hat(W) {
  const r = BODY_RX, H = BODY_H;
  const cL = TRI_CX - TRI_W / 2;
  const cR = TRI_CX + TRI_W / 2;
  return [
    `M 0,0`,                                  // 왼쪽 평평 시작
    `H ${W - r}`,                             // 본체 위 직선
    `A ${r},${r} 0 0,1 ${W},${r}`,            // 오른쪽 위 캡슐
    `V ${H - r}`,
    `A ${r},${r} 0 0,1 ${W - r},${H}`,        // 오른쪽 아래 캡슐
    `H ${cR}`, `L ${TRI_CX},${H + TRI_H}`, `L ${cL},${H}`,  // 아래 ▽
    `H 0`, 'z',                               // 왼쪽 평평 마감
  ].join(' ');
}

필드 카탈로그

contents 배열에 들어가는 항목 5종. 각 항목은 measureContent()로 폭을 미리 측정해 본체 SVG 너비를 결정한 뒤 renderContent()가 위치를 따라 그린다.

type용도시각
text정적 글자흰색 굵은 NanumGothic 13px
input숫자/문자 입력칸흰 둥근 박스 + 검정 글씨 + 회색 stroke
dropdown선택 박스카테고리 darken 색 둥근 박스 + 흰 글씨 + ▾ 화살표
icon인라인 아이콘ICONS[name] SVG 통째 inline
block자식 블록 끼움 (재귀 nesting)자식 SVG를 renderBlock() 재귀 호출 후 <g transform>으로 배치

데이터 모델

한 블록은 { shape, category, icon?, marker?, contents, statements?, middleContents? }로 표현된다. contents는 필드 객체 배열이며, contents[i].type === 'block'이면 block 필드에 또 다른 블록 객체가 들어가 재귀 nesting된다.

{
  "shape": "stack",
  "category": "MOVING",
  "contents": [
    { "type": "text",  "value": "x 좌표를" },
    { "type": "block", "block": {
        "shape": "reporter",
        "category": "VARIABLE",
        "contents": [{ "type": "text", "value": "걷는속도" }]
    }},
    { "type": "text",  "value": "만큼 바꾸기" }
  ]
}

cblock·cblock_double은 statements 필드로 자식 stack 배열을 받는다.

{
  "shape": "cblock",
  "category": "FLOW",
  "contents": [{ "type": "input", "value": "10" }, { "type": "text", "value": "번 반복하기" }],
  "statements": [
    { "shape": "stack", "category": "MOVING", "contents": [/* ... */] },
    { "shape": "stack", "category": "LOOKS",  "contents": [/* ... */] }
  ]
}

사용 패턴

패턴 1 — HTML data-blocks (권장)

<script src="assets/entry-blocks/renderer.js" defer></script>

<div class="entry-code" data-blocks='[
  {"shape":"hat","category":"START","icon":"play",
   "contents":[{"type":"text","value":"시작하기 버튼을 클릭했을 때"}]},
  {"shape":"stack","category":"MOVING","note":"매 프레임 5칸씩",
   "contents":[
     {"type":"text","value":"x 좌표를"},
     {"type":"input","value":"5"},
     {"type":"text","value":"만큼 바꾸기"}
   ]}
]'></div>

EntryBlocks.render('#my-target', {
  shape: 'cblock',
  category: 'FLOW',
  contents: [{type:'input',value:'10'},{type:'text',value:'번 반복하기'}],

    { shape:'stack', category:'MOVING',
      contents:[{type:'text',value:'x 좌표를'},{type:'input',value:'1'},{type:'text',value:'만큼 바꾸기'}] }
  ],
});

알려진 한계

한계
  • c-block 자식 statements 안의 note 누락: 새 시스템에서 note는 .entry-code-row 단위(최상위 블록)에만 표시. c-block 내부 자식 stack의 note는 SVG 내부라 누락됨. 1편 묶음 3·4, 3편 묶음 4 등 일부 학습 보조 텍스트 손실
  • 복합 reporter 단순화: "걷는속도 × 0.85" 같은 계산 블록 nesting은 단순 텍스트로 표현. 정확한 reporter+reporter 구조까지는 처리하지 않음
  • 모양 픽셀 일치 안 됨: production 에디터의 SVG path를 직접 가져온 게 아니라 디자인을 보고 자체 작성. 둥글기·세모 크기 등은 시각적으로 비슷하지만 픽셀 단위로는 완전히 일치하지 않음
  • 스크롤 시 좌우 잘림: 매우 깊이 중첩된 c-block(3편 묶음 4 서브스텝)의 폭이 모바일 viewport를 초과할 수 있음. 가로 스크롤 또는 줄바꿈 정책 미정
